## Часть 0. Как работает сеть

> Нельзя настраивать веб-сервер, не понимая, что происходит между браузером и сервером.
> Эту часть на собеседованиях спрашивают чаще, чем сам nginx.

### Клиент-серверная модель

Весь веб — это диалог двух ролей:

- **Клиент** — тот, кто **инициирует** запрос (браузер, мобильное приложение, `curl`).
- **Сервер** — тот, кто **ждёт** запросов и отвечает на них (nginx, твой Go-бэкенд).

Сервер не может первым «постучаться» в браузер. Он пассивно слушает и отвечает.
Это фундамент: nginx — это программа, которая **слушает** порт и отвечает на HTTP-запросы.

### Модель OSI и TCP/IP: где живёт nginx

Сеть разбита на **уровни (layers)**. Каждый уровень решает свою задачу и не знает,
что делают соседние. Тебе не нужно знать все 7 уровней OSI наизусть, но нужно твёрдо
понимать **4 ключевых** из практической модели TCP/IP:

```
┌─────────────────────────────────────────────────────────────┐
│  L7  Application   HTTP, HTTPS, DNS, gRPC   ← тут работает    │
│                                               nginx и бэкенд   │
├─────────────────────────────────────────────────────────────┤
│  L4  Transport     TCP, UDP                 ← порты, надёжная  │
│                                               доставка         │
├─────────────────────────────────────────────────────────────┤
│  L3  Network       IP                       ← адресация,       │
│                                               маршрутизация     │
├─────────────────────────────────────────────────────────────┤
│  L2/L1  Link/Phys  Ethernet, Wi-Fi          ← провода, кадры   │
└─────────────────────────────────────────────────────────────┘
```

Запомни главное разделение, его любят на собесах:

- **L4 (транспортный)** оперирует **IP:порт**. Он не знает про URL, куки, заголовки.
  Балансировщик L4 просто пересылает TCP-пакеты.
- **L7 (прикладной)** понимает **HTTP**: URL, заголовки, методы, куки.
  nginx — это в первую очередь **L7 reverse proxy**: он читает `GET /api/todos`
  и решает, куда это отправить. Именно поэтому он может маршрутизировать по путям.

### IP-адрес, порт, сокет

- **IP-адрес** — адрес машины в сети. `127.0.0.1` (он же `localhost`) — это «сам себя».
- **Порт** — номер «двери» на машине (0–65535). На одной машине много сервисов,
  порт говорит, к какому именно ты стучишься.
  - `80` — HTTP по умолчанию, `443` — HTTPS по умолчанию.
  - В твоём проекте бэкенд слушает `8080`, фронт (vite preview) — `4173`.
- **Сокет** — конкретная пара `IP:порт`, например `127.0.0.1:8080`. Это то, что
  «слушает» сервер. Двух процессов на одном `IP:порт` быть не может — отсюда ошибка
  `address already in use`, которую ты обязательно однажды увидишь.

### DNS: как имя превращается в IP

Ты пишешь `google.com`, но компьютеры общаются по IP. **DNS** — это телефонная книга
интернета, которая переводит имя → IP.

```
браузер ──"где google.com?"──► DNS-резолвер ──► 142.250.x.x
```

Внутри Docker Compose работает **встроенный DNS**: имя сервиса (`backend`, `frontend`)
резолвится в IP контейнера. Именно поэтому в конфиге можно писать `proxy_pass http://backend:8080`,
а не хардкодить IP — Docker сам подставит нужный адрес. Это ты используешь в Части 6.

### TCP: надёжное соединение и «тройное рукопожатие»

HTTP работает поверх **TCP** — протокола, который гарантирует, что байты дойдут
в правильном порядке и без потерь. Прежде чем отправить хоть один байт данных,
клиент и сервер устанавливают соединение через **three-way handshake**:

```
Клиент                          Сервер
  │  ── SYN ──────────────────►   │   "хочу соединиться"
  │  ◄────────────── SYN-ACK ──   │   "ок, я готов, ты готов?"
  │  ── ACK ──────────────────►   │   "готов, поехали"
  │                               │
  │  ═══ теперь идут HTTP-данные ═══
```

Почему это важно для nginx: каждое новое соединение — это накладные расходы
(3 пакета туда-обратно + для HTTPS ещё TLS-рукопожатие). Поэтому существует
**keep-alive** — переиспользование одного TCP-соединения для нескольких запросов.
В Части 7 ты увидишь `keepalive` в блоке `upstream` — это ровно про экономию
на рукопожатиях между nginx и бэкендом.

### Анатомия HTTP-запроса и ответа

Открой то, что реально летит по проводу. HTTP — это **текст** (в HTTP/1.1).

**Запрос:**
```http
GET /api/todos HTTP/1.1        ← метод, путь, версия
Host: localhost                 ← обязательный заголовок: какой сайт нужен
User-Agent: curl/8.0            ← кто спрашивает
Accept: application/json        ← что готов принять
                                ← пустая строка = конец заголовков
                                ← (для POST тут было бы тело)
```

**Ответ:**
```http
HTTP/1.1 200 OK                 ← версия, код статуса, текст
Content-Type: application/json  ← что за данные в теле
Content-Length: 87              ← сколько байт в теле
Server: nginx/1.27.0            ← кто ответил

{"status":"success","data":[…]} ← тело ответа
```

Ключевые элементы, которые ты будешь трогать в nginx:

- **Метод**: `GET`, `POST`, `PUT`, `PATCH`, `DELETE`. Твой бэкенд использует
  `GET/POST/PATCH/DELETE` для todo.
- **Заголовок `Host`** — критичен. По нему nginx понимает, какой из сайтов
  (`server`-блоков) на одном IP тебе нужен. Это база виртуального хостинга (Часть 4).
- **Коды статуса** — выучи группы наизусть, это классический вопрос:
  - `1xx` — информационные (редко).
  - `2xx` — успех (`200 OK`, `201 Created`, `204 No Content`).
  - `3xx` — редиректы (`301` навсегда, `302` временно, `304 Not Modified` — кэш).
  - `4xx` — ошибка клиента (`400`, `401` не авторизован, `403` запрещено,
    `404` не найдено, `429` слишком много запросов).
  - `5xx` — ошибка сервера (`500`, `502 Bad Gateway` — nginx не достучался
    до бэкенда, `503`, `504 Gateway Timeout`). **`502` и `504` ты будешь ловить
    постоянно при настройке proxy — научись их читать.**

## Часть 1. Что такое веб-сервер и что такое nginx

### Что вообще делает «веб-сервер»

Веб-сервер — это программа, которая:
1. **Слушает** TCP-порт (обычно 80/443).
2. **Принимает** HTTP-запросы.
3. **Отдаёт** ответ — либо файл с диска (статика), либо результат работы другой
   программы (проксирование на бэкенд).

Всё. Никакой магии. `nginx`, `Apache`, `Caddy`, встроенный сервер в твоём Go —
все они делают это. Разница в архитектуре и удобстве.

### Две роли nginx, которые надо различать

Это ключевой концепт. nginx умеет быть двумя разными вещами (часто одновременно):

- **Web server (origin)** — сам отдаёт файлы с диска: HTML, CSS, JS, картинки.
  Так ты будешь раздавать собранный React (Часть 4).
- **Reverse proxy** — не обслуживает запрос сам, а **пересылает** его на другой сервер
  (твой Go-бэкенд) и возвращает клиенту его ответ. Так ты подключишь `/api` (Часть 6).

> **Forward proxy vs reverse proxy** (любимый вопрос на собесе):
> - **Forward proxy** стоит на стороне **клиента** и прячет клиента от сервера
>   (VPN, корпоративный прокси).
> - **Reverse proxy** стоит на стороне **сервера** и прячет бэкенды от клиента.
>   Клиент думает, что говорит с одним сервером, а за ним может быть 10 бэкендов.
>   nginx — это reverse proxy.

### Зачем вообще reverse proxy перед бэкендом

Ты мог бы открыть Go-бэкенд напрямую в интернет. Почему так не делают:

- **Единая точка входа.** Фронт и `/api` на одном домене и порту → нет проблем с CORS,
  один TLS-сертификат, одна точка для логов и лимитов.
- **TLS-терминация.** nginx расшифровывает HTTPS, бэкенд работает по простому HTTP
  и не думает о сертификатах.
- **Балансировка.** Один вход → несколько инстансов бэкенда (Часть 7).
- **Защита и буферизация.** Rate limiting, отсечение медленных клиентов, отдача статики
  без нагрузки на бэкенд.
- **Гибкость.** Можно переписывать пути, кэшировать, отдавать заглушку при падении бэкенда.

### Архитектура nginx: почему он быстрый

Это **главная теоретическая тема** для собеса. Разберём подробно.

**Старая модель (классический Apache prefork):** на каждое соединение — отдельный
процесс или поток. 10 000 клиентов → 10 000 потоков → огромный расход памяти
и постоянные переключения контекста. Это упирается в «проблему C10k».

**Модель nginx: event-driven, асинхронная, неблокирующая.**

```
        ┌──────────────┐
        │   master      │  запущен от root: читает конфиг,
        │   process     │  открывает порты, управляет воркерами
        └──────┬───────┘
               │ форкает
     ┌─────────┼─────────┐
     ▼         ▼         ▼
 ┌───────┐ ┌───────┐ ┌───────┐
 │worker │ │worker │ │worker │   обычно = число CPU-ядер
 │(1 ядро)│ │(1 ядро)│ │(1 ядро)│  каждый обслуживает ТЫСЯЧИ
 └───────┘ └───────┘ └───────┘   соединений в одном потоке
```

- **master-процесс** ничего не обслуживает. Он читает конфиг, биндит порты
  (для этого нужны root-права) и управляет воркерами. Когда ты делаешь
  `nginx -s reload`, master запускает **новые** воркеры с новым конфигом
  и мягко гасит старые — **без обрыва соединений**. Это zero-downtime reload.
- **worker-процессы** делают всю работу. Их обычно столько же, сколько ядер CPU
  (`worker_processes auto;`). Каждый воркер — **однопоточный**, но обслуживает
  тысячи соединений одновременно за счёт **event loop**.

**Как один поток тянет тысячи соединений?** Через механизм ОС `epoll` (Linux).
Вместо «поток ждёт данные от клиента и простаивает», nginx говорит ядру:
«разбуди меня, когда на любом из этих 5000 сокетов появятся данные». Пока данных нет —
воркер занимается другими соединениями. Никто никого не блокирует. Это и есть
**неблокирующий event-driven I/O**. Отсюда крошечное потребление памяти
и способность держать десятки тысяч соединений.

## Часть 2. Установка и первый запуск

Есть два пути. Пройди **оба** — на реальной работе встречаются оба.

### Путь А: нативная установка (чтобы пощупать руками)

**Manjaro / Arch (твоя система):**
```bash
sudo pacman -S nginx
sudo systemctl start nginx        # запустить
sudo systemctl enable nginx       # автозапуск при загрузке
sudo systemctl status nginx       # проверить, что жив
```

Открой в браузере `http://localhost` — увидишь страницу «Welcome to nginx!».
Поздравляю, ты только что поднял веб-сервер. Он отдаёт файл
`/usr/share/nginx/html/index.html` — обычную статику.

Ключевые команды CLI (выучи наизусть, ими пользуешься каждый день):
```bash
nginx -v          # версия
nginx -V          # версия + с какими модулями собран (важно на собесе)
nginx -t          # проверить конфиг на ошибки, НЕ применяя (делай ВСЕГДА перед reload)
nginx -T          # проверить + вывести весь итоговый конфиг
sudo nginx -s reload   # перечитать конфиг без остановки (zero-downtime)
sudo nginx -s stop     # немедленная остановка
sudo nginx -s quit     # мягкая остановка (доработать текущие запросы)
sudo systemctl reload nginx   # то же, что -s reload, но через systemd
```

> **Правило номер один:** перед каждым `reload` делай `nginx -t`. Если в конфиге
> ошибка, `reload` её отвергнет, но лучше знать заранее. `-t` спас миллионы продакшенов.

## Часть 3. Анатомия конфига

Весь nginx настраивается **текстовыми конфигами**. Пойми их структуру — и всё
остальное станет читаемым.

### Директивы и контексты

Конфиг состоит из **директив** двух видов:

- **Простые директивы** — `имя значение;` (заканчиваются точкой с запятой):
  ```nginx
  worker_processes auto;
  listen 80;
  ```
- **Блочные директивы (контексты)** — `имя { … }`, содержат другие директивы:
  ```nginx
  http {
      server {
          listen 80;
      }
  }
  ```

**Контекст** — это область видимости. Директива действует внутри своего контекста
и обычно **наследуется** во вложенные. Иерархия контекстов:

```
main (сам файл, самый верхний уровень)
│   worker_processes, user, pid, error_log
│
├── events { }          ← настройки соединений
│       worker_connections
│
└── http { }            ← весь HTTP: тут живёт 99% твоих настроек
    │   gzip, log_format, include, keepalive_timeout
    │
    ├── upstream { }     ← группа бэкендов для балансировки (Часть 7)
    │
    └── server { }       ← один «виртуальный сайт»
        │   listen, server_name, ssl_certificate, access_log
        │
        └── location { } ← правила для конкретных путей (Часть 5)
                proxy_pass, root, try_files
```

Запомни: `location` живёт внутри `server`, `server` — внутри `http`,
`events` и `http` — на верхнем уровне. Перепутаешь уровень — получишь ошибку
вида `"proxy_pass" directive is not allowed here`.

### Наследование

Директивы «текут вниз». Если задать `gzip on;` в `http`, он включится во всех `server`
и `location` внутри — пока какой-нибудь вложенный контекст не переопределит.
Это как переменные с областью видимости в коде.

### include — как конфиг разбивают на файлы

В реальности никто не пишет один гигантский файл. Используют `include`:

```nginx
http {
    include /etc/nginx/conf.d/*.conf;   # подтянуть все .conf из папки
}
```

Отсюда два популярных подхода к организации:
- **`conf.d/`** — просто складываешь файлы `*.conf`, все они инклудятся. Так делает
  официальный Docker-образ. **Мы пойдём этим путём** — он проще.
- **`sites-available/` + `sites-enabled/`** — конфиги всех сайтов лежат в `available`,
  а «включённые» — это симлинки в `enabled`. Так делает Debian/Ubuntu из коробки.
  Удобно включать/выключать сайты, не удаляя конфиг.

### Пишем свой главный конфиг

Создай `nginx/nginx.conf` в проекте с таким содержимым. **Прочитай каждый комментарий** —
это скелет любого nginx-конфига:

```nginx
# ===== main-контекст =====
worker_processes auto;          # воркеров = числу ядер CPU (см. Часть 1)

events {
    worker_connections 1024;    # макс. соединений на один воркер
}

http {
    include       /etc/nginx/mime.types;   # таблица "расширение → Content-Type"
    default_type  application/octet-stream; # тип по умолчанию, если не нашли

    sendfile        on;         # эффективная отдача файлов (см. Часть 10)
    keepalive_timeout  65;      # сколько держать неактивное соединение

    # Подключаем конфиги наших сайтов из conf.d/
    include /etc/nginx/conf.d/*.conf;
}
```

Обрати внимание: тут **нет** блока `server`. Он будет в `conf.d/default.conf` —
его мы напишем в следующей части. Так конфиг «движка» отделён от конфига «сайта».

## Часть 4. Раздача статики

Первый **настоящий** сайт: nginx как web server, отдающий файлы. Раздадим собранный React.

### Минимальный server-блок

Создай `nginx/conf.d/default.conf`:

```nginx
server {
    listen 80;                      # слушать порт 80
    server_name localhost;          # для какого домена (имени) этот блок

    root /usr/share/nginx/html;     # корневая папка со статикой
    index index.html;               # какой файл отдавать для "/"

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

Разберём построчно:

- **`listen 80`** — на каком порту (и опционально IP) принимать соединения.
  Можно `listen 443 ssl;`, `listen 8080;` и т.д.
- **`server_name localhost`** — по какому значению заголовка `Host` этот блок
  срабатывает. Так один nginx обслуживает много доменов на одном IP
  (**виртуальный хостинг**). `server_name mchs-simulator.ru www.mchs-simulator.ru;` — несколько имён.
- **`root`** — папка на диске, от которой считаются пути. Запрос `/css/app.css`
  → файл `/usr/share/nginx/html/css/app.css`.
- **`index`** — что отдать, когда запросили папку (`/` → `index.html`).

### root vs alias — классическая путаница (и вопрос на собесе)

```nginx
# root: путь запроса ДОБАВЛЯЕТСЯ к root
location /static/ {
    root /var/www;
    # запрос /static/app.js  →  файл /var/www/static/app.js
}

# alias: путь запроса ЗАМЕНЯЕТСЯ на alias
location /static/ {
    alias /var/www/assets/;
    # запрос /static/app.js  →  файл /var/www/assets/app.js
}
```

Правило: **`root` приклеивает URL к пути, `alias` подменяет совпавшую часть location.**
Перепутаешь — получишь 404 на ровном месте.

### try_files и SPA — почему это важно для React

```nginx
location / {
    try_files $uri $uri/ /index.html;
}
```

`try_files` перебирает варианты **слева направо** и отдаёт первый существующий:
1. `$uri` — файл ровно по запрошенному пути (например, `/assets/app.js` — есть файл, отдаём).
2. `$uri/` — то же, но как папку.
3. `/index.html` — **fallback**, если ничего не нашли.

Зачем fallback для React: у SPA роутинг на клиенте. Если пользователь открыл
`localhost/todos/42`, такого файла на диске нет — но нам нужно отдать `index.html`,
чтобы React-роутер сам разобрался с путём. Без `try_files … /index.html` ты получишь
`404` при перезагрузке страницы на любом под-маршруте. **Это классический баг деплоя SPA.**

## Часть 5. location: матчинг и приоритеты

`location` — сердце маршрутизации nginx. Когда приходит запрос, nginx выбирает **ровно один**
location-блок. Правила выбора **не** «сверху вниз» — и это ловушка на собеседованиях.

### Типы location

```nginx
location = /exact      { }   # ТОЧНОЕ совпадение пути
location ^~ /assets/   { }   # префикс с приоритетом (стоп-регэксп)
location ~ \.php$      { }   # регэксп, ЧУВСТВИТЕЛЬНЫЙ к регистру
location ~* \.(jpg|png)$ { } # регэксп, НЕчувствительный к регистру
location /api/         { }   # обычный префикс
location /             { }   # префикс "/", матчит вообще всё
```

### Порядок выбора (выучи наизусть)

nginx выбирает location по такому алгоритму:

1. **`=` (точное совпадение)** — если есть, побеждает **сразу**. Поиск окончен.
2. Ищется **самый длинный совпавший префиксный** location (обычные и `^~`).
3. Если этот самый длинный префикс — с модификатором **`^~`**, он побеждает,
   регэкспы **не проверяются**.
4. Иначе проверяются **регэкспы** (`~` / `~*`) **в порядке их следования в файле**.
   Первый совпавший регэксп побеждает.
5. Если ни один регэксп не совпал — берётся тот самый длинный префикс из шага 2.

Короткая формула для памяти:
> **точное `=` → префикс `^~` → регэксп (по порядку) → самый длинный обычный префикс.**

### Пример, который всё проясняет

```nginx
server {
    location = / {
        # ТОЛЬКО "/" и ничего больше
        return 200 "точный корень\n";
    }
    location / {
        # всё остальное
        return 200 "любой префикс\n";
    }
    location /api/ {
        # всё, что начинается с /api/
        return 200 "api-префикс\n";
    }
    location ~ \.json$ {
        # любой путь, кончающийся на .json
        return 200 "json-регэксп\n";
    }
}
```

- `GET /` → «точный корень» (сработал `=`).
- `GET /about` → «любой префикс» (самый длинный подходящий префикс — `/`).
- `GET /api/todos` → «api-префикс» (префикс `/api/` длиннее, чем `/`).
- `GET /api/data.json` → «json-регэксп»! Хотя префикс `/api/` совпал, **регэксп имеет
  приоритет над обычным префиксом**. Вот эта строчка и валит людей на собесе.

`return 200 "текст";` — удобный способ тестировать матчинг, не заводя реальные файлы.

## Часть 6. Reverse proxy

Кульминация курса: свяжем всё вместе. nginx будет **одновременно** раздавать React-статику
и **проксировать** `/api` на Go-бэкенд. Единый вход, никакого CORS.

### proxy_pass — основа проксирования

```nginx
location /api/ {
    proxy_pass http://localhost:8080;
}
```

`proxy_pass` говорит: «не обслуживай запрос сам, а перешли его вот на этот адрес
и верни клиенту ответ оттуда». nginx становится посредником.

### Тонкость с завершающим слэшем в proxy_pass (частый баг)

Это ловушка, которую спрашивают на собесе. Наличие/отсутствие `/` в конце `proxy_pass`
меняет путь, который уходит на бэкенд:

```nginx
# БЕЗ слэша: путь передаётся ЦЕЛИКОМ, включая /api/
location /api/ {
    proxy_pass http://localhost:8080;
    # запрос /api/todos  →  бэкенд получает /api/todos
}

# СО слэшем: часть, совпавшая с location, ОТРЕЗАЕТСЯ и заменяется
location /api/ {
    proxy_pass http://localhost:8080/;
    # запрос /api/todos  →  бэкенд получает /todos   (без /api!)
}
```

Твой Go-бэкенд регистрирует роуты **с префиксом `/api`** (`router.Group("/api")`
в `app.go`). Значит, тебе нужен вариант **БЕЗ** завершающего слэша — чтобы `/api/todos`
дошёл до бэкенда как есть. Запомни: **слэш в конце `proxy_pass` = «отрезать префикс location».**

### Проксирующие заголовки — обязательная гигиена

Когда nginx проксирует запрос, бэкенд по умолчанию видит **nginx** как клиента,
а не реального пользователя. Чтобы бэкенд знал правду, пробрасывают заголовки:

```nginx
location /api/ {
    proxy_pass http://localhost:8080;

    proxy_set_header Host              $host;              # исходный домен из запроса
    proxy_set_header X-Real-IP         $remote_addr;       # реальный IP клиента
    proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for; # цепочка прокси
    proxy_set_header X-Forwarded-Proto $scheme;            # http или https
}
```

Зачем каждый:
- **`Host`** — без него бэкенд увидит `Host: localhost:8080` (адрес из `proxy_pass`),
  а не исходный домен. Ломает логику, завязанную на домен.
- **`X-Real-IP` / `X-Forwarded-For`** — иначе в логах бэкенда IP всех клиентов будет
  одинаковым (IP nginx). Для аналитики, банов, rate limit на бэке — критично.
- **`X-Forwarded-Proto`** — бэкенд за nginx работает по HTTP и «не знает», что снаружи
  был HTTPS. Этот заголовок сообщает исходную схему (важно для генерации ссылок и редиректов).

`$host`, `$remote_addr`, `$scheme` — это **встроенные переменные** nginx. Их десятки;
эти четыре — самые ходовые.

### Финальный конфиг: статика + проксирование

Вот он, целевой `nginx/conf.d/default.conf` всего курса:

```nginx
server {
    listen 80;
    server_name localhost;

    # --- API: проксируем на Go-бэкенд ---
    location /api/ {
        proxy_pass http://backend:8080;    # "backend" — имя сервиса в compose

        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # --- Всё остальное: статика React ---
    location / {
        root /usr/share/nginx/html;
        index index.html;
        try_files $uri $uri/ /index.html;   # SPA-fallback (Часть 4)
    }
}
```

Обрати внимание: имя `backend` вместо `localhost:8080`. Внутри Docker-сети
`localhost` — это сам контейнер nginx, а не бэкенд. Резолвить имя сервиса в IP
контейнера будет встроенный DNS Docker (см. Часть 0).

## Часть 7. upstream и балансировка

Пока за nginx один бэкенд. В проде их несколько — для отказоустойчивости и нагрузки.
Тут появляется блок `upstream`.

### Блок upstream

```nginx
# в контексте http { }
upstream backend_pool {
    server backend1:8080;
    server backend2:8080;
    server backend3:8080;
}

server {
    location /api/ {
        proxy_pass http://backend_pool;   # проксируем на ГРУППУ, не на один хост
    }
}
```

`upstream` — именованная группа серверов. `proxy_pass` ссылается на неё по имени.
nginx сам распределяет запросы между членами группы.

### Алгоритмы балансировки

```nginx
upstream backend_pool {
    # round-robin — по умолчанию, ничего писать не надо:
    # запросы по кругу: 1 → 2 → 3 → 1 → 2 → 3 …

    # least_conn — тому, у кого меньше активных соединений
    #   (хорошо, когда запросы разной длительности):
    # least_conn;

    # ip_hash — клиент всегда попадает на один и тот же сервер
    #   (нужно для sticky-сессий):
    # ip_hash;

    server backend1:8080 weight=3;   # весами задаётся доля трафика
    server backend2:8080;            # weight=1 по умолчанию
    server backend3:8080 backup;     # запасной: включается, если остальные легли
}
```

- **round-robin** — просто по очереди. Дефолт, подходит для большинства случаев.
- **least_conn** — на наименее загруженный. Лучше при неравномерных по времени запросах.
- **ip_hash** — «липкость» по IP клиента. Нужно, если сессия хранится в памяти
  конкретного инстанса (в идеале сессии выносят в Redis и липкость не нужна).
- **`weight=N`** — мощному серверу даёшь больше трафика.
- **`backup`** — резерв на случай падения основных.

### Пассивные health-check'и

nginx (в бесплатной версии) делает **пассивные** проверки: если сервер вернул ошибку
или не ответил, он временно исключается из ротации.

```nginx
upstream backend_pool {
    server backend1:8080 max_fails=3 fail_timeout=30s;
    # 3 неудачи за 30с → сервер считается упавшим на 30с
}
```

> Активные health-check'и (nginx сам периодически «пингует» бэкенды по расписанию)
> есть только в платном **nginx Plus**. На собесе полезно знать эту границу
> free/Plus. В опенсорсе активные проверки часто добавляют через сторонние модули.

### keepalive к бэкендам

Помнишь стоимость TCP-рукопожатия из Части 0? Между nginx и бэкендами тоже не стоит
на каждый запрос открывать новое соединение:

```nginx
upstream backend_pool {
    server backend1:8080;
    keepalive 32;              # держать до 32 idle-соединений к бэкендам
}

server {
    location /api/ {
        proxy_pass http://backend_pool;
        proxy_http_version 1.1;          # keepalive требует HTTP/1.1
        proxy_set_header Connection "";  # убрать заголовок Connection: close
    }
}
```

## Часть 8. Production-настройки

То, что отличает «работает на ноуте» от «работает в проде». Каждую тему спрашивают
на собеседованиях уровня middle+.

### Логирование

nginx пишет два лога:
- **access.log** — каждый обслуженный запрос.
- **error.log** — ошибки и диагностика (уровни: `debug`, `info`, `warn`, `error`, `crit`).

Формат access-лога настраивается через `log_format`:

```nginx
http {
    log_format main '$remote_addr - [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    'rt=$request_time uct="$upstream_connect_time" '
                    'urt="$upstream_response_time"';

    access_log /var/log/nginx/access.log main;
}
```

Ключевые переменные для дебага производительности:
- **`$request_time`** — полное время обработки запроса nginx'ом.
- **`$upstream_response_time`** — сколько отвечал бэкенд. Если `$request_time` большой,
  а `$upstream_response_time` маленький — тормозит сеть/клиент, а не бэкенд. **Это
  первое, что смотрят при разборе «почему медленно».**

### gzip-сжатие

Сжимаем текстовые ответы перед отправкой — меньше трафика, быстрее загрузка:

```nginx
http {
    gzip on;
    gzip_types text/plain text/css application/json
               application/javascript text/xml application/xml;
    gzip_min_length 1024;      # не жать совсем мелкое (накладные расходы не окупятся)
    gzip_comp_level 5;         # 1 (быстро) … 9 (сильно, но дорого по CPU)
}
```

> Картинки (jpg, png) и так сжаты — их жать бесполезно, только CPU потратишь.
> Для максимума используют **brotli** (нужен модуль), но gzip — базовый минимум.

### Кэширование статики (заголовки для браузера)

Скажем браузеру держать ассеты в кэше, чтобы не качать их каждый раз:

```nginx
location /assets/ {
    root /usr/share/nginx/html;
    expires 30d;                              # заголовок Expires + Cache-Control
    add_header Cache-Control "public, immutable";
}
```

Работает, потому что Vite добавляет хэш в имена файлов (`index-a1b2c3.js`).
Поменялся файл → поменялось имя → браузер скачает новый, а старые лежат в кэше.

### Проксирующий кэш (nginx кэширует ответы бэкенда)

nginx может кэшировать ответы бэкенда и отдавать их сам, разгружая бэк:

```nginx
http {
    proxy_cache_path /var/cache/nginx keys_zone=api_cache:10m max_size=100m inactive=60m;

    server {
        location /api/ {
            proxy_pass http://backend:8080;
            proxy_cache api_cache;
            proxy_cache_valid 200 10s;         # кэшировать успешные ответы на 10с
            add_header X-Cache-Status $upstream_cache_status;  # HIT/MISS в ответе
        }
    }
}
```

`$upstream_cache_status` в ответе покажет `MISS` (сходили на бэк) или `HIT`
(отдали из кэша). Кэшировать нужно осторожно — только то, что не зависит от юзера.

### Rate limiting — защита от перегрузки и абьюза

Ограничиваем частоту запросов с одного IP:

```nginx
http {
    # зона памяти: считаем запросы по IP, лимит 10 запросов/сек
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

    server {
        location /api/ {
            limit_req zone=api_limit burst=20 nodelay;
            proxy_pass http://backend:8080;
        }
    }
}
```

- **`rate=10r/s`** — базовая скорость: 10 запросов в секунду с одного IP.
- **`burst=20`** — разрешить всплеск до 20 запросов сверх нормы (они встают в очередь).
- **`nodelay`** — обслужить всплеск сразу, а не размазывать по времени.
- Превысил лимит → nginx отдаёт **`503`** (или `429`, если настроить). Помнишь коды
  из Части 0?

Похожая директива `limit_conn` ограничивает число **одновременных соединений** с IP.

### Таймауты и буферы

```nginx
location /api/ {
    proxy_pass http://backend:8080;

    proxy_connect_timeout 5s;    # сколько ждать установки соединения с бэком
    proxy_send_timeout   10s;    # таймаут на отправку запроса бэку
    proxy_read_timeout   30s;    # сколько ждать ответа бэка (частый источник 504)
}
```

`504 Gateway Timeout`, который ты, возможно, ловил, — это чаще всего превышение
`proxy_read_timeout`: бэкенд не ответил вовремя.

## Часть 9. HTTPS и TLS

Финальный production-навык. Без HTTPS сегодня в интернет никого не пускают.

### Что такое TLS и зачем

**TLS** (наследник SSL) шифрует канал между клиентом и сервером. Он даёт три вещи:
1. **Шифрование** — трафик нельзя прочитать по пути.
2. **Целостность** — данные нельзя незаметно подменить.
3. **Аутентификация** — сертификат подтверждает, что сервер тот, за кого себя выдаёт.

`https://` = HTTP поверх TLS, порт `443`.

### Сертификаты и центры сертификации (CA)

- **Сертификат** — файл, который связывает домен с публичным ключом и подписан
  доверенным **центром сертификации (CA)**.
- Браузер доверяет набору корневых CA. Если сертификат подписан ими — замочек зелёный.
- **Let's Encrypt** — бесплатный CA, автоматизировал выдачу. Стандарт индустрии.
- **Self-signed** — сертификат, который ты подписал сам. Браузер будет ругаться
  («не доверяю»), но для локальной учёбы годится.

### TLS-рукопожатие (упрощённо, но для собеса достаточно)

```
Клиент                                   Сервер
  │ ── ClientHello ──────────────────►    │  "поддерживаю такие шифры,
  │    (версии TLS, шифры)                 │   вот случайное число"
  │ ◄──────────── ServerHello + сертификат │  "выбрал шифр, вот мой
  │                                        │   сертификат с публ. ключом"
  │  [проверяет сертификат у CA]           │
  │ ── обмен ключами ────────────────►     │  договариваются об общем
  │ ◄──────────────────────────────────    │  симметричном ключе сессии
  │                                        │
  │ ═══ дальше всё шифруется симметрично ═══
```

Ключевая идея: асимметричная криптография (медленная) используется только чтобы
**договориться** об общем симметричном ключе, а дальше данные шифруются быстрым
симметричным шифром. Это и спрашивают: «почему не шифровать всё асимметрично?» —
потому что это дорого; асимметрия нужна лишь для безопасного обмена ключом.

### TLS-терминация на nginx

Типичный паттерн: **nginx расшифровывает HTTPS, а с бэкендом говорит по обычному HTTP.**
Бэкенд вообще не знает про сертификаты. Это называется **TLS termination**.

```
браузер ──HTTPS/443──► nginx ──HTTP/8080──► backend
         (шифрованно)          (внутри доверенной сети)
```

### Настройка HTTPS в nginx

Мы **не будем** генерировать self-signed сертификат локально — раз у тебя есть
сервер и домен **mchs-simulator.ru**, сразу настроим настоящий сертификат от
Let's Encrypt прямо на сервере (Часть 9.5). Здесь только разберём теорию: **из чего
состоит** HTTPS-блок nginx.

Конфиг из двух server-блоков — один редиректит HTTP на HTTPS, второй обслуживает TLS:

```nginx
# HTTP → редирект на HTTPS
server {
    listen 80;
    server_name mchs-simulator.ru;
    return 301 https://$host$request_uri;   # 301 = постоянный редирект
}

# HTTPS
server {
    listen 443 ssl;
    server_name mchs-simulator.ru;

    # пути к сертификату Let's Encrypt (появятся после выпуска в Части 9.5)
    ssl_certificate     /etc/letsencrypt/live/mchs-simulator.ru/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/mchs-simulator.ru/privkey.pem;

    ssl_protocols TLSv1.2 TLSv1.3;          # старьё (TLS 1.0/1.1) выключено

    # HSTS: браузер сам будет ходить только по HTTPS
    add_header Strict-Transport-Security "max-age=31536000" always;

    location /api/ {
        proxy_pass http://backend:8080;
        proxy_set_header Host              $host;
        proxy_set_header X-Forwarded-Proto $scheme;   # тут будет "https"
    }

    location / {
        root /usr/share/nginx/html;
        try_files $uri $uri/ /index.html;
    }
}
```

Порт `443` и папку с сертификатами пробрасывают в compose так (для Let's Encrypt
это том `./certbot/conf:/etc/letsencrypt` — детали в Части 9.5):
```yaml
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./certbot/conf:/etc/letsencrypt:ro
```

> **В реальном проде** self-signed заменяют на **Let's Encrypt** через `certbot`
> или через связку с `acme.sh`. Certbot сам получает и **автоматически продлевает**
> сертификат (он живёт 90 дней). Принцип конфига тот же — меняются только пути
> к сертификату.
>
> 👉 Полный практический разбор для реального домена в Docker (certbot, webroot,
> автопродление, правильное проксирование с TLS-терминацией) — в **Части 9.5**.

## Часть 9.5. Боевой HTTPS на своём домене через certbot

Часть 9 дала self-signed сертификат для локалки (браузер ему не доверяет). Теперь —
**настоящий** сертификат от **Let's Encrypt**, которому доверяют все браузеры,
бесплатно и с автопродлением. Это то, что реально стоит в проде. Делаем для сетапа
в Docker Compose (nginx в контейнере) — именно так, как у тебя.

### Что понадобится (предусловия)

1. **Домен**, указывающий на IP твоего сервера. В DNS-настройках домена должна быть
   **A-запись** (для IPv4) — например `mchs-simulator.ru → 203.0.113.10`. Проверь, что
   применилось:
   ```bash
   dig +short mchs-simulator.ru        # должен вернуть IP твоего сервера
   ```
   DNS обновляется не мгновенно (до нескольких часов) — дождись, иначе выпуск упадёт.

   > В командах и конфиге ниже используется только `mchs-simulator.ru`. Если захочешь
   > ещё и `www.mchs-simulator.ru` — заведи для него A-запись и добавь его **и** в
   > `-d www.mchs-simulator.ru` у certbot, **и** в `server_name` обоих server-блоков.

2. **Открытые порты 80 и 443** на сервере (firewall / security group облака).
   Порт 80 нужен обязательно, даже если сайт только по HTTPS — через него идёт
   проверка владения доменом (см. ниже).

3. **Проект уже поднимается** через docker-compose на этом сервере.

### Как Let's Encrypt проверяет, что домен твой (ACME http-01)

Центр сертификации не выдаёт сертификат кому попало — он должен убедиться, что ты
**контролируешь домен**. Механизм называется **ACME challenge**, самый частый вариант —
**http-01**:

```
1. certbot говорит Let's Encrypt: "хочу сертификат на mchs-simulator.ru"
2. Let's Encrypt: "докажи. Положи файл с таким-то токеном по адресу
   http://mchs-simulator.ru/.well-known/acme-challenge/<токен>"
3. certbot кладёт этот файл в webroot-папку, которую отдаёт nginx
4. Let's Encrypt заходит по http://mchs-simulator.ru/.well-known/... и читает токен
5. Совпало → домен твой → выдаётся сертификат (в /etc/letsencrypt/live/mchs-simulator.ru/)
```

Вот почему **порт 80 должен быть открыт и отдавать `/.well-known/acme-challenge/`
по обычному HTTP без редиректа** — это канал проверки. Запомни это, к нему вернёмся
в подводных камнях.

### Проблема «курицы и яйца» в Docker

Тонкий момент, о который все спотыкаются: чтобы nginx запустился с `listen 443 ssl`,
ему нужны **файлы сертификата**. Но чтобы **получить** сертификат, нужен запущенный
nginx, отдающий ACME-challenge на порту 80. Замкнутый круг.

Решаем в два шага: **сначала поднимаем nginx только по HTTP** (он умеет отдавать
challenge и без сертификата), **получаем сертификат**, и **только потом** добавляем
HTTPS-блок.

### Шаг 1. Volumes и сервис certbot в docker-compose

Добавь в `docker-compose.yaml` две общие папки (webroot для challenge и хранилище
сертификатов) и сервис `certbot`:

```yaml
services:
  nginx:
    image: nginx:1.27-alpine
    container_name: todo_nginx
    ports:
      - "80:80"
      - "443:443"                                   # добавили HTTPS-порт
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./nginx/conf.d:/etc/nginx/conf.d:ro
      - ./certbot/www:/var/www/certbot:ro           # сюда certbot кладёт challenge
      - ./certbot/conf:/etc/letsencrypt:ro          # сюда — сами сертификаты
    depends_on:
      - backend
      - frontend
    restart: unless-stopped

  certbot:
    image: certbot/certbot
    container_name: todo_certbot
    volumes:
      - ./certbot/www:/var/www/certbot:rw           # те же папки, но на запись
      - ./certbot/conf:/etc/letsencrypt:rw
    # по умолчанию простаивает; запускаем вручную командой ниже
    entrypoint: "/bin/sh -c 'trap exit TERM; while :; do sleep 6h & wait $${!}; certbot renew; done'"
```

Обрати внимание: nginx монтирует те же папки **на чтение** (`:ro`), а certbot —
**на запись** (`:rw`). Так они обмениваются файлами: certbot пишет — nginx читает.

### Шаг 2. Bootstrap-конфиг (только HTTP, для получения сертификата)

Временно замени `nginx/conf.d/default.conf` на HTTP-only вариант — он умеет отдавать
challenge и пока проксирует сайт по обычному HTTP (замени `mchs-simulator.ru` на свой домен):

```nginx
server {
    listen 80;
    server_name mchs-simulator.ru;    # ТВОЙ реальный домен, не localhost!

    # ACME-challenge: отдаём файлы, которые кладёт certbot. БЕЗ редиректа!
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    # пока сертификата нет — просто проксируем сайт по HTTP
    location /api/ {
        proxy_pass http://backend:8080;
    }
    location / {
        proxy_pass http://frontend:4173;
    }
}
```

Подними стек и проверь, что домен открывается по HTTP:
```bash
docker compose up -d
curl -I http://mchs-simulator.ru          # ожидаешь ответ от твоего сайта
```

### Шаг 3. Получаем сертификат

**Сначала — в тестовом режиме `--staging`** (у Let's Encrypt строгий лимит: ~5 боевых
сертификатов на домен в неделю; на этапе отладки легко его выжечь, а staging — без
лимитов, но сертификат «ненастоящий»):

```bash
docker compose run --rm certbot certonly \
  --webroot -w /var/www/certbot \
  -d mchs-simulator.ru \
  --email petruhenvladimir@gmail.com --agree-tos --no-eff-email \
  --staging
```

Разбор флагов:
- `certonly` — только получить сертификат, ничего не настраивать (конфиг мы пишем сами).
- `--webroot -w /var/www/certbot` — метод webroot: класть challenge в эту папку
  (её отдаёт nginx из Шага 2).
- `-d ...` — домены (можно несколько; первый станет именем папки в `live/`).
- `--staging` — тестовый CA.

Если увидел `The dry run was successful` / успешный выпуск staging — значит цепочка
работает. Теперь получаем **боевой** сертификат (убираем `--staging`, добавляем
`--force-renewal`, чтобы перезаписать staging-версию):

```bash
docker compose run --rm certbot certonly \
  --webroot -w /var/www/certbot \
  -d mchs-simulator.ru \
  --email petruhenvladimir@gmail.com --agree-tos --no-eff-email \
  --force-renewal
```

Сертификат появится в `./certbot/conf/live/mchs-simulator.ru/`:
- `fullchain.pem` — сертификат + промежуточные (его указываем в `ssl_certificate`),
- `privkey.pem` — приватный ключ (`ssl_certificate_key`).

### Шаг 4. Финальный конфиг: редирект + HTTPS + проксирование

Теперь, когда сертификат есть, переписываем `nginx/conf.d/default.conf` в боевой вид —
**два server-блока**: HTTP (challenge + редирект) и HTTPS (основной сайт):

```nginx
# --- HTTP :80 — только challenge и редирект на HTTPS ---
server {
    listen 80;
    server_name mchs-simulator.ru;

    # ВАЖНО: challenge оставляем на HTTP без редиректа — иначе продление сломается
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    # всё остальное — жёстко на HTTPS
    location / {
        return 301 https://$host$request_uri;
    }
}

# --- HTTPS :443 — основной сайт ---
server {
    listen 443 ssl;
    http2 on;                                  # HTTP/2 поверх TLS — быстрее
    server_name mchs-simulator.ru;

    # Сертификаты от Let's Encrypt
    ssl_certificate     /etc/letsencrypt/live/mchs-simulator.ru/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/mchs-simulator.ru/privkey.pem;

    # Современные настройки TLS
    ssl_protocols       TLSv1.2 TLSv1.3;       # старьё (1.0/1.1) выключено
    ssl_prefer_server_ciphers off;             # для TLS1.3 выбор шифра за клиентом
    ssl_session_cache   shared:SSL:10m;        # кэш сессий → меньше рукопожатий
    ssl_session_timeout 1d;

    # HSTS: браузер запомнит ходить только по HTTPS (год). Включай, когда уверен,
    # что HTTPS работает стабильно — откатить сложно (браузеры кэшируют).
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # --- API: проксируем на бэкенд ---
    location /api/ {
        limit_req zone=api_limit burst=20 nodelay;
        proxy_pass http://backend:8080;

        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;   # тут будет "https"
    }

    # --- Фронтенд ---
    location / {
        proxy_pass http://frontend:4173;

        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

Применяем без даунтайма:
```bash
docker compose exec nginx nginx -t          # проверить синтаксис
docker compose exec nginx nginx -s reload   # применить
```

Открой `https://mchs-simulator.ru` — зелёный замочек, доверенный сертификат. HTTP
автоматически редиректит на HTTPS.

### Как ПРАВИЛЬНО проксировать с SSL (то, что часто спрашивают)

Ключевая схема называется **TLS-терминация** и работает так:

```
браузер ──HTTPS (443, шифрованно)──► nginx ──HTTP (8080, открыто)──► backend
         внешний, недоверенный канал         внутренняя Docker-сеть,
         → обязателен TLS                     доверенная → TLS не нужен
```

Три правила правильного SSL-проксирования:

1. **TLS расшифровывается на nginx, до бэкенда идёт обычный HTTP.** Бэкенд внутри
   Docker-сети (`backend:8080`) не занимается сертификатами вообще — это дёшево и
   просто. Сеть между контейнерами доверенная, шифровать её незачем.

2. **Обязательно пробрасывай `X-Forwarded-Proto $scheme`.** Бэкенд получает запрос по
   HTTP и «думает», что снаружи тоже был HTTP. Этот заголовок сообщает ему, что
   изначально был **https**. Без него бэкенд, генерируя абсолютные ссылки или редиректы,
   выдаст `http://...` — и получится «смешанный контент» или редирект-петля. Твой Go
   за прокси должен доверять именно этому заголовку, а не своему соединению.

3. **`Host $host`** — чтобы бэкенд видел реальный домен (`mchs-simulator.ru`), а не
   `backend:8080`.

> **Когда шифруют и до бэкенда (end-to-end TLS)?** В моделях zero-trust или когда
> бэкенд в другой сети/дата-центре. Тогда `proxy_pass https://backend:8443;` и бэкенд
> тоже с сертификатом. Для одной Docker-сети это избыточно — стандарт тут TLS-терминация.

> **WebSocket за HTTPS.** Если появится WebSocket (`wss://`), в его `location` добавь
> проброс апгрейда: `proxy_set_header Upgrade $http_upgrade;` и
> `proxy_set_header Connection "upgrade";` + `proxy_http_version 1.1;`. Для обычного
> REST-API (как твой todo) это не нужно.

### Автопродление (сертификат живёт 90 дней)

Let's Encrypt выдаёт сертификаты на **90 дней** — это заставляет автоматизировать
продление. У тебя это уже настроено сервисом `certbot` из Шага 1: его `entrypoint`
каждые 6 часов вызывает `certbot renew`. `renew` продлевает сертификат, только если
до истечения осталось меньше 30 дней (иначе ничего не делает) — так что дёргать часто
безопасно.

Один нюанс: после продления **nginx должен перечитать** новый сертификат. Простое
решение — периодически перезагружать nginx. Замени его запуск на самоперезагружающийся:

```yaml
  nginx:
    # ... остальное как выше ...
    command: >
      /bin/sh -c 'while :; do sleep 6h & wait $${!}; nginx -s reload; done & nginx -g "daemon off;"'
```

Проверить продление вручную (без реального выпуска):
```bash
docker compose run --rm certbot renew --dry-run
```

### Подводные камни (частые ошибки)

- **`/.well-known/acme-challenge/` попал под редирект на HTTPS** → продление падает
  (Let's Encrypt ходит по HTTP). В HTTP-блоке этот `location` должен стоять **выше**
  редиректа и отдавать webroot — как в конфиге Шага 4.
- **`server_name` = `localhost`** вместо реального домена → сертификат не выпустится.
  Домен в `server_name` и в `-d` у certbot должны совпадать.
- **Сжёг лимит боевых сертификатов** (5/неделю на домен), отлаживаясь без `--staging`.
  Всегда сначала `--staging`.
- **Порт 80/443 закрыт файрволом** → challenge не проходит или сайт недоступен.
  Проверь `sudo ss -tlnp | grep -E ':80|:443'` и правила облака.
- **DNS ещё не обновился** → Let's Encrypt резолвит домен в старый/чужой IP.
  Дождись `dig +short mchs-simulator.ru` = твой IP.

### Как один воркер держит тысячи соединений: event loop и epoll

Вернёмся к Части 1 и углубимся. Воркер nginx — это цикл событий:

```
while (true) {
    events = epoll_wait();       // ядро отдаёт список "готовых" сокетов
    for (event : events) {       // где появились данные / место для записи
        handle(event);           // обработать не блокируясь и идти дальше
    }
}
```

- **Блокирующая модель** (поток на соединение): поток вызвал `read()` и **завис**,
  пока не придут данные. 10k соединений → 10k «зависших» потоков → память и context switches.
- **Неблокирующая модель nginx**: `read()` возвращает управление сразу. Воркер спрашивает
  у ядра (`epoll` на Linux, `kqueue` на BSD/macOS) «на каких из моих тысяч сокетов
  уже есть работа» и обрабатывает только их. Один поток — тысячи соединений, почти
  без простоя. Это и есть решение **проблемы C10k** (10 000 одновременных соединений).

### Директивы производительности, которые надо знать

```nginx
worker_processes auto;        # = числу ядер CPU

events {
    worker_connections 1024;  # соединений на воркер
    # multi_accept on;        # принимать несколько соединений за раз
}

http {
    sendfile    on;    # отдавать файлы напрямую из ядра (zero-copy), минуя userspace
    tcp_nopush  on;    # с sendfile: слать заголовки одним полным пакетом
    tcp_nodelay on;    # для keep-alive: не задерживать мелкие пакеты (отключить Nagle)
}
```

- **`sendfile`** — файл отправляется из дискового кэша ядра прямо в сокет, не копируясь
  в память процесса nginx. Это **zero-copy**, огромный выигрыш на раздаче статики.
- **`tcp_nopush`** (работает с `sendfile`) — накопить и отправить заголовки+начало файла
  одним заполненным пакетом, а не россыпью мелких.
- **`tcp_nodelay`** — отключает алгоритм Нейгла для keep-alive соединений, чтобы мелкие
  ответы (например, API) уходили сразу, а не ждали накопления.

Теоретический потолок соединений: `worker_processes × worker_connections`. Но помни:
на **проксировании** каждый клиентский запрос держит **два** соединения (клиент↔nginx
и nginx↔бэкенд), так что реальная ёмкость вдвое меньше.

### graceful reload: как обновить конфиг без даунтайма

Когда ты делаешь `nginx -s reload`:
1. master проверяет новый конфиг (если ошибка — старый продолжает работать, reload отклонён).
2. master запускает **новые** воркеры с новым конфигом.
3. **Старые** воркеры перестают принимать новые соединения, но **дорабатывают** текущие.
4. Когда старые доработали — master их гасит.

Итог: конфиг применился, **ни один запрос не оборвался**. Это то, за что nginx любят
в проде, и частый вопрос: «как задеплоить новый конфиг без даунтайма?».

## Что дальше

Ты прошёл путь от «что такое порт» до production-ready reverse proxy с HTTPS,
балансировкой и защитой. Дальше углубляйся по мере надобности:

- **Официальная документация** — `nginx.org/en/docs/`. Читается тяжело, но это
  источник истины. Учись находить в ней директивы.
- **Модули**: `stream` (L4-проксирование TCP/UDP, например для баз данных),
  `njs` (скрипты на JS внутри nginx), `headers-more`.
- **Альтернативы для кругозора**: Caddy (автоматический HTTPS из коробки),
  Traefik (нативная интеграция с Docker/Kubernetes), Envoy (L7-прокси для service mesh).
- **Kubernetes**: `Ingress`-контроллер на nginx — как всё это выглядит в оркестрации.
- **Наблюдаемость**: экспорт метрик nginx в Prometheus, дашборды в Grafana.
